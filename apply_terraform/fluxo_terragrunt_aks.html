Visão Geral do Projeto

Este projeto de Infraestrutura como Código (IaC) foi desenvolvido para provisionar de forma automatizada e padronizada um cluster Azure Kubernetes Service (AKS) integrado ao Argo CD para GitOps. Para que este repositório possa ser utilizado com segurança e eficácia por qualquer membro da organização, ele deve seguir padrões rigorosos de design modular, segurança, portabilidade e resiliência.

Nota Técnica Importante: O código atualmente faz uso de Providers Dinâmicos. Os provedores de kubernetes e helm utilizam atributos gerados pelo módulo do AKS (como host, tokens e CA certificados). Entenda a fundo este fluxo nas seções abaixo antes de aplicar em produção.

Passo a Passo do Fluxo de Provisionamento

Fase 1 Grupo de Recursos (Azure Resource Group)

O Terraform inicializa instanciando o módulo resource_group. Ele gera o contêiner lógico no Azure sob o qual todos os outros recursos residirão. A saída desse módulo fornece o nome e a localização física essenciais para a vinculação da rede e do cluster.

Inputs Utilizados
  • resource_group_name ("rg-aks-dev")
  • location ("eastus")
Outputs Exportados
  • name (Nome do RG criado)
  • location (Região de deploy)

Fase 2 Infraestrutura de Rede (VNet e Subnet)

O módulo network é acionado logo após a confirmação de existência do Resource Group. Ele cria a Virtual Network (VNet) e uma Subnet dedicada. Essa rede é essencial porque o AKS será configurado em modo de integração nativa de rede do Azure (Azure CNI), fazendo com que cada Pod ganhe um IP real pertencente a essa subnet.

Inputs Utilizados
  • vnet_name, vnet_address_space ("10.0.0.0/8")
  • subnet_name, subnet_prefixes ("10.240.0.0/16")
Outputs Exportados
  • vnet_id (Identificador de Rede)
  • subnet_id (Utilizado pelo AKS Node Pool)

Fase 3 Cluster Kubernetes (Azure AKS)

O módulo aks provisiona o cluster gerenciado. Ele vincula o pool de nós padrão (System Node Pool) diretamente na subnet criada anteriormente. O cluster é configurado com a feature do OIDC Issuer habilitada (oidc_issuer_enabled = true) que possibilita o Workload Identity (prática recomendada para ligar ServiceAccounts do K8s a identidades do Azure sem usar segredos estáticos).

Inputs Utilizados
  • node_count (1 nó padrão), node_vm_size ("Standard_B2s")
  • subnet_id (Obtido dinamicamente do módulo de rede)
Outputs Exportados
  • Credentials: host, client_certificate, client_key, cluster_ca_certificate

Fase 4 bootstrapping de Aplicação (Argo CD Helm)

Com o cluster AKS online, os outputs de certificado e host alimentam diretamente os provedores do kubernetes e helm em tempo de execução. O Terraform então cria o namespace argocd e instala o Helm chart oficial do Argo CD (versão default 7.3.1). Isso encerra o ciclo de deploy conectando a infraestrutura à engine de GitOps.

Inputs Utilizados
  • enable_argocd (true/false)
  • argocd_chart_version ("7.3.1")
  • argocd_helm_values (Opções personalizadas para o Argo CD)
Outputs Exportados
  • N/A (Módulo focado apenas em deploy de software)

Auditoria de Boas Práticas (Best Practices Check)

Abaixo está uma avaliação detalhada do código atual sob a perspectiva de segurança, resiliência, compartilhamento e padrões corporativos.

Seguida Organização de Código
Modularização e Abstração

O código divide perfeitamente os domínios: Resource Group, Network, AKS, e Argo CD estão isolados. Qualquer usuário do repositório pode reusar partes ou o conjunto completo.

Impacto: Facilita a manutenção do código e a replicação de ambientes idênticos.

Violada Gerenciamento de Estado
Estado Local (terraform.tfstate local)

Atualmente, o state file do Terragrunt é gravado localmente no diretório do operador. Se múltiplos usuários rodarem esse código, eles criarão recursos duplicados, sobrescreverão o state ou perderão o controle das instâncias existentes no Azure.

Ação Crítica Necessária: Configurar um backend do Azure Storage Account com State Locking ativado.

Risco Potencial Dependências de Inicialização
Acoplamento Dinâmico de Provedores K8s/Helm

Definir os blocos de provedor do Kubernetes/Helm puxando atributos gerados no próprio plano (como module.aks.host) é uma prática comum para demos simples, mas que pode causar erros em ambientes do zero (Fresh Apply). O Terraform precisa planejar os recursos do Argo CD antes do cluster existir, e sem acesso ao host, pode acusar falha de conexão logo de início.

Recomendação Corporativa: Separar a infra de rede e AKS em um diretório (ex: 01-infra) e a entrega das ferramentas de cluster em outro (ex: 02-bootstrap-apps).

Melhorável Alta Disponibilidade
Node Pool Rígido e Sem Autoscaling

O pool padrão está configurado com número fixo de nós (node_count = 1) e máquina pequena (Standard_B2s). Cargas críticas e o próprio Argo CD concorrerão por recursos limitados. Em produção, é essencial separar pools de sistema de pools de aplicação e habilitar o autoscaling.

Recomendação: Habilitar autoscaling com min_count e max_count e dimensionar a VM para pelo menos a série D (ex: Standard_D2s_v5).

Vulnerabilidade Rede e Acesso
API Control Plane Exposta Publicamente

A API do Kubernetes está aberta globalmente para a internet. Para um template reutilizável de nível profissional, o cluster deve suportar a ativação de IPs públicos autorizados (api_server_authorized_ip_ranges) ou ser privado (private_cluster_enabled = true).

Ação: Parametrizar a restrição de IPs de rede autorizados no módulo AKS.

Melhorias Recomendadas (Como Refatorar)

Use os templates abaixo para atualizar os arquivos correspondentes e elevar a qualidade do seu Terraform para nível Enterprise Ready.

Crie este arquivo na raiz do repositório 
# backend.tf - Centraliza o estado para que qualquer membro do time compartilhe a mesma infra
terraform {
  backend "azurerm" {
    resource_group_name  = "infra-shared-resources-rg"
    storage_account_name = "tfstateaksenterprise"
    container_name       = "tfstate"
    key                  = "aks-cluster/terraform.tfstate"
    use_azuread_auth     = true # Autenticação mais segura via RBAC Entra ID
  }
}
cluster_aks/variables.tf (Evitar defaults sensíveis e documentar variáveis) 
variable "resource_group_name" {
  type        = string
  description = "Nome do grupo de recursos no Azure"
  # Remova o default rígido para forçar o usuário a configurar seu ambiente explicitamente
}

variable "allowed_ip_ranges" {
  type        = list(string)
  description = "Lista de IPs (CIDR) com permissão para acessar a API Server do AKS"
  default     = ["0.0.0.0/0"] # Altere para o IP corporativo ou de sua VPN em prod
}
cluster_aks/modules/aks/terragrunt.hcl (Habilitando Autoscaling e Segurança) 
resource "azurerm_kubernetes_cluster" "aks" {
  name                = var.cluster_name
  location            = var.location
  resource_group_name = var.resource_group_name
  dns_prefix          = var.dns_prefix

  default_node_pool {
    name                = "system"
    vnet_subnet_id      = var.subnet_id
    vm_size             = "Standard_D2s_v5" # Recomendado para produção
    
    # Configurações de Auto-scaling essenciais
    enable_auto_scaling = true
    min_count           = 1
    max_count           = 3
  }

  api_server_authorized_ip_ranges = var.allowed_ip_ranges # Proteção da API

  identity {
    type = "SystemAssigned"
  }

  oidc_issuer_enabled       = true
  workload_identity_enabled = true # Prática moderna de segurança
}

Guia Prático do Operador (Comandos)

Abaixo está a trilha de comandos necessária para que qualquer operador prepare o ambiente e execute este código Terraform de forma segura.

1. Autenticação Inicial no Azure

Garanta o login correto e selecione a assinatura de destino para evitar provisionar na conta errada.

az login
az account show --query "{name:name, id:id}"
# Se necessário, troque de assinatura:
az account set --subscription "ID-DA-SUA-ASSINATURA"
2. Ciclo de Execução do Terragrunt

Navegue até a pasta correspondente e inicie as fases:

cd cluster_aks
terragrunt run-all init      # Baixa os providers (azurerm, kubernetes, helm)
terraform validate  # Checa a sintaxe
terragrunt run-all plan      # Analisa o impacto antes de efetivar
terragrunt run-all apply     # Digite "yes" para aprovar o deploy
3. Conectar kubectl ao Cluster AKS

Obtenha o contexto de acesso local para interagir com o cluster e verificar o Argo CD:

az aks get-credentials --resource-group rg-aks-dev --name terragrunt-aks-dev
kubectl get namespaces
kubectl get pods -n argocd
